# 开发工作流

## 分支策略

| 分支 | 用途 |
|------|------|
| `master` | 稳定分支（每个版本完成后合入） |
| `dev` | 开发集成分支 |
| `v0.x` | 版本分支（从 dev 拉出，完成后合回 dev） |

**流程**：
1. **设计阶段**在 dev 分支进行 → spec 变更审核通过后 commit
2. 从 dev 拉 `v0.x` 分支 → 进入开发
3. 开发完成 + 验收通过 → 合并 `v0.x → dev`
4. 版本稳定后合并 `dev → master`

## 版本开发流程

每个版本按顺序经历三个阶段：**设计 → 开发 → 验收**。版本串行推进。

### 阶段一：设计

进入新版本开发前，必须先完成设计并通过审核。

#### Step 1：需求梳理

1. 阅读 `spec/requirements.md`，确认本版本要实现的功能
2. 阅读 `spec/versions.md`，确认本版本在路线图中的位置和前置依赖
3. 列出本版本要交付的核心能力（1-2 句话概括）
4. 确认上游版本是否已完成

#### Step 2：任务拆分

1. 将版本目标拆分为 3-6 个子任务
2. **拆分原则**：

| 原则 | 说明 |
|------|------|
| 单文件优先 | 每个任务只新建/修改 1-3 个文件 |
| 可独立验证 | 每个任务有自己的验收标准，不依赖同一版本的其他任务结果才能验证 |
| 有意义的 commit | 每个任务完成后的 commit message 能清晰描述做了什么 |
| 后端→前端 | 后端任务完成后，前端任务才有稳定的 API 可调 |
| 不超前实现 | 只实现当前版本范围内的功能，不做预判性开发 |

3. 确定任务之间的依赖顺序：
   ```
   v0.x-1 (无依赖) → v0.x-2 (依赖1) → v0.x-3 (依赖2) → v0.x-4 (依赖1, 可并行)
   ```
4. 在 `spec/versions/v0.x.md` 顶部写出子任务清单

#### Step 3：任务细化

按以下模板逐个填写每个子任务，**不要遗漏任何段落**：

```markdown
## v0.x-N 任务名 [ ]

### 目标
（1-2 句话，从用户/调用方角度描述这个任务完成后能做什么）

### 依赖
- v0.x-M（前置子任务）
- 或：无依赖

### 改动文件

| 操作 | 文件 | 说明 |
|------|------|------|
| 新建 | `server/routers/xxx.py` | xxx API 路由 |
| 修改 | `server/main.py` | 注册新路由 |

### 实现
（关键代码片段。不需要完整源码，但要足够清晰，让开发者无需再读其他文档就能编码）

### 测试

（后端：pytest 测试代码；前端：手动验证步骤）

**测试文件顶部必须包含行为注释**：
```python
# 测试行为：
# - test_xxx_normal: 正常输入返回预期结果
# - test_xxx_edge: 边界输入正确处理
```

### 验收标准
- [ ] 可验证的条件 1（具体、可测量）
- [ ] 可验证的条件 2
```

**填写规则**：
- 验收标准必须是可验证的（"能跑通" 不行，"curl 返回 200 + body 含 status:ok" 才行）
- 改动文件清单必须穷尽（漏写文件会导致开发时不知道该改哪个）
- 实现代码片段必须可运行（import 路径、类名、函数签名与项目结构一致）

#### Step 4：版本验收标准定义

在版本文件末尾添加整体验收条件：

```markdown
## v0.x 版本验收

### 整体验收
- [ ] 全量 pytest 通过，零失败
- [ ] （前端版本）浏览器手动验证全部页面可正常加载和交互
- [ ] （有存储版本）SQLite 数据写入和查询正确
- [ ] 本版本引入的核心链路可端到端走通

### 不包含
- xxx（本版本明确不做的功能）
- xxx
```

#### Step 5：架构一致性检查

1. 打开 `spec/architecture.md`，逐项检查：
   - 本版本的改动文件是否在目录结构中出现（新建文件则确认目录结构是否需要更新）
   - 本版本引入的 API 端点是否在端点清单中出现（新端点则追加）
   - 本版本引入的数据模型是否与已有模型冲突
2. 打开 `spec/versions.md`，检查：
   - 依赖关系图是否仍正确
   - Mock 策略是否符合当前版本所处阶段
3. 如有不一致，先更新相关 spec 文件

#### Step 6：设计审核

1. **自查**：
   - 每个子任务的六段式（目标/依赖/改动文件/实现/测试/验收标准）是否完整
   - 改动文件清单是否穷尽
   - 验收标准是否可验证
   - 测试代码是否包含行为注释
2. **展示给审核人**：
   - 说明本版本的目标和与上下游版本的关系
   - 逐个子任务说明目标、改动范围、验收标准
   - 确认架构一致性检查结果
   - 确认不包含的范围
3. **审核要点**：

| 维度 | 检查项 |
|------|--------|
| 拆分 | 任务数 3-6 个、每个可独立验证、依赖顺序合理 |
| 完整性 | 六段式无遗漏、改动文件穷尽、验收标准可验证 |
| 架构 | 与 architecture.md 一致、无跨层依赖 |
| Mock | 如版本需要 mock，确保 mock 数据与真实接口契约一致 |
| 范围 | 无超出 MVP 范围的功能、不含项明确 |

4. **审核不通过** → 记录修改意见 → 修改 spec → 重新审核
5. **审核通过** → 进入 Step 7 提交

#### Step 7：提交设计

1. 更新 `spec/versions.md` 中本版本状态为 `[~]`
2. commit spec 变更：
   ```bash
   git add spec/versions/v0.x.md spec/versions.md spec/architecture.md
   git commit -m "docs: v0.x 设计完成"
   ```
3. 创建版本分支：
   ```bash
   git checkout -b v0.x
   ```
4. 进入阶段二（开发）

### 阶段二：开发

每个子任务按以下步骤执行，完成后才能开始下一个子任务（串行）。

#### Step 1：任务启动

1. 打开 `spec/versions/v0.x.md`，找到当前子任务
2. 阅读本任务的 **目标 / 依赖 / 改动文件 / 实现 / 测试 / 验收标准**
3. 确认前置依赖任务已完成（`[x]`）
4. 更新子任务清单中本任务状态：`[ ]` → `[~]`
5. 如任务修改已有文件，先 `git status` 确认工作区干净

#### Step 2：编写测试（后端）

> 后端代码先写测试再实现。前端不写自动化测试，直接写组件。

1. 在 `spec/versions/v0.x.md` 的「测试」段落中找到测试代码
2. 按 spec 中的测试目录路径创建测试文件（如 `server/tests/unit/test_xxx.py`）
3. **在文件顶部以注释形式描述本文件测试的行为**：
   ```python
   # 测试行为：
   # - test_xxx_returns_yyy: 输入合法参数时，返回正确结果
   # - test_xxx_raises_on_invalid: 输入非法参数时，抛出 ValueError
   # - test_xxx_edge_case: 空输入时返回默认值
   ```
4. 将测试代码写入文件，确保 import 路径正确
5. 运行测试，预期 **失败**（红灯 — 实现还没写）
6. 检查失败原因：
   - 模块不存在 → 先创建空模块/空函数让 import 通过
   - 断言错误 → 正常，进入 Step 3

```bash
python -m pytest server/tests/unit/test_xxx.py -v
```

#### Step 3：测试行为审核

> 在编写实现代码之前，审核测试代码是否正确描述了预期行为。

1. **自查**：
   - 测试文件顶部的行为注释是否完整、准确
   - 每个测试是否对应 spec「验收标准」中的一条
   - 测试是否覆盖了正常路径 + 边界情况 + 异常情况
2. **展示给审核人**：
   - 逐条过测试行为注释
   - 确认测试逻辑与 spec 的验收标准一致
   - 确认没有遗漏验收条件
3. **审核要点**：

| 维度 | 检查项 |
|------|--------|
| 行为注释 | 是否清晰描述了每个 test 验证什么 |
| 覆盖度 | 是否覆盖了所有验收标准 |
| 边界 | 是否包含空输入、非法输入、极值等边界测试 |
| 可读性 | 测试命名是否见名知义 |

4. **审核不通过** → 修改测试 → 重新审核
5. **审核通过** → 进入 Step 4 编写实现

#### Step 4：实现代码

1. 按「改动文件」表格确认要新建/修改的文件清单
2. 按「实现」段落中的代码片段编写代码
3. **代码规范**：
   - Python：类型注解（`def foo(x: int) -> str:`），异步优先
   - React：函数组件 + Hooks，TypeScript 类型
   - 命名：Python `snake_case`，React `PascalCase` 组件 / `camelCase` 函数
   - import 排序：标准库 → 第三方 → 项目内
4. 自查：改动文件是否与 spec 一致、无遗漏 import、无 hardcode 密钥

#### Step 5：运行测试 / 手动验证

**后端**：
```bash
python -m pytest server/tests/unit/test_xxx.py -v   # 本任务测试
python -m pytest server/tests/ -v                     # 确认无回归
```

**前端**：
```bash
cd client && npm run dev
# 浏览器 localhost:5173，对照验收标准逐条验证
```

#### Step 6：对照验收标准

1. 回到 spec 本任务的「验收标准」段落，逐条确认
2. 后端：确认测试覆盖了所有验收标准
3. 前端：确认浏览器手动验证了所有条件

#### Step 7：人工审核

> 每个子任务完成后、提交前，必须经过人工审核。

1. **自查**（开发者）：
   - `git diff` 逐文件检查所有变更
   - 确认没有多余的调试代码（print、console.log）
   - 确认没有 hardcode 的密钥/路径
   - 确认改动文件清单与 spec 一致
2. **展示给审核人**：
   - 说明本任务的目标和完成情况
   - 展示改动文件列表和关键代码
   - 运行测试或前端效果演示
   - 逐条过验收标准
3. **审核要点**：

| 维度 | 检查项 |
|------|--------|
| 代码 | 逻辑正确、无副作用、命名清晰 |
| Spec | 改动文件与 spec 一致、验收标准全部满足 |
| 测试 | 测试覆盖了核心路径和边界情况 |
| 安全 | 无密钥泄漏、无 SQL 注入、输入已校验 |
| 规范 | 符合命名规范、import 排序正确 |

4. **审核不通过** → 记录修改意见 → 回到 Step 4 修改 → 重新 Step 5-6 → 再次审核
5. **审核通过** → 进入 Step 8 提交

#### Step 8：提交

1. 更新 `spec/versions/v0.x.md`：子任务 `[~]` → `[x]`
2. `git add` 所有变更（代码 + 测试 + spec）
3. commit，遵循 [Conventional Commits](commit.md)：
   ```bash
   git commit -m "feat: v0.1-1 项目初始化 + 目录结构"
   ```
4. **每个子任务一个独立 commit**

#### 循环

完成一个子任务后，从 Step 1 开始执行下一个子任务，直到本版本所有子任务 `[x]`。

#### 开发规则

| 规则 | 说明 |
|------|------|
| 禁止跨任务修改 | 只改当前任务的改动文件 |
| 修改须来自 spec | 需改 spec 未列出的文件时，先更新 spec 再改代码 |
| 测试不过不提交 | 后端所有测试必须通过才 commit |
| 遇阻即停 | 任务无法推进时记录问题后停止，不强行绕过 |

### 阶段三：验收

版本全部子任务完成后，进入验收阶段。验收通过才能合入 dev。

#### Step 1：全量测试

1. 运行全部后端测试，确认零失败：
   ```bash
   cd server && python -m pytest tests/ -v
   ```
2. 如有测试失败，回到对应子任务修改，直到全部通过
3. 前端任务：启动服务，在浏览器中走一遍完整用户流程

#### Step 2：版本验收标准检查

1. 打开 `spec/versions/v0.x.md`，找到「版本验收」段落
2. 逐条确认每个验收条件是否满足
3. 每确认一条，勾选 `[ ]` → `[x]`
4. 有未满足的 → 记录缺失项 → 确定修复方案 → 创建修复任务

#### Step 3：全链路手动验证

> 模拟真实用户操作，端到端走通本版本引入的功能。

1. 启动完整环境（后端 + 前端）
2. 按照 `spec/overview.md` 中的核心交互流程操作
3. 记录操作过程中的任何卡顿、报错、非预期行为
4. **验证范围**取决于当前版本：

#### Step 4：Bug 记录

1. 将验收中发现的 bug 记录到 `spec/bug/v0.x.md`：
   ```markdown
   # v0.x Bug 记录

   | # | 描述 | 严重度 | 状态 |
   |---|------|--------|------|
   | 1 | 空输入导致 500 | 中 | [ ] |
   | 2 | 前端白屏 | 高 | [x] |
   ```
2. **严重度定义**：

| 级别 | 含义 | 处理 |
|------|------|------|
| 高 | 核心功能不可用、数据丢失 | **必须修复**才能合入 |
| 中 | 边界情况异常、体验瑕疵 | 记录，下版本修复 |
| 低 | UI 微调、文案优化 | 记录，择机修复 |

3. 高级别 bug 全部修复后，重新执行 Step 1-2

#### Step 5：Spec 同步

> 确认 spec 文档与当前实现一致。

1. 检查 `spec/overview.md`：
   - 用户故事是否与已实现功能匹配
   - 核心交互流程是否可走通
2. 检查 `spec/architecture.md`：
   - 目录结构是否与实际一致（新增/删除的文件）
   - API 端点是否与 routers 中一致
   - 数据模型是否与 models 中一致
3. 检查 `spec/requirements.md`：
   - 是否有需求因实现过程而变更（如 scope 调整）
4. 如有不一致，更新 spec 文档并 commit

#### Step 6：合入

1. 确认所有条件满足：
   - [ ] 全部测试通过
   - [ ] 版本验收标准全部 `[x]`
   - [ ] 无高级别未修复 bug
   - [ ] Spec 文档已同步
2. 更新 `spec/versions.md` 中本版本状态为 `[x]`
3. 提交合入 commit：
   ```bash
   git add spec/versions.md spec/versions/v0.x.md
   git commit -m "chore: v0.x 验收通过"
   ```
4. 合入分支：
   ```bash
   git checkout dev
   git merge v0.x
   ```
5. 删除版本分支（可选）：`git branch -d v0.x`

## 任务状态标记

| 标记 | 含义 |
|------|------|
| `[ ]` | 未开始 |
| `[~]` | 进行中 |
| `[x]` | 已完成 |

## 运行测试

```bash
# 全部后端测试
cd server && python -m pytest tests/ -v

# 单个文件
python -m pytest tests/unit/test_question_gen.py -v

# watch 模式（需要 pytest-watch）
ptw tests/ -- -v

# 前端开发服务器
cd client && npm run dev

# 后端开发服务器
cd server && uvicorn main:app --reload
```
